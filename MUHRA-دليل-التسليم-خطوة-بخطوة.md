## جزء ب — Git ورفع الكود (كل مرة تعدّل)

### 1) افتح GitHub Desktop

1. **File → Add Local Repository** إن لم يكن مضافًا، واختر مجلد:  
   `C:\Users\al7us\Desktop\MUHRA JEWELRY`
2. تأكد الفرع: **main**.

### 2) قبل أي عمل: اسحب آخر شيء من الإنترنت

- اضغط **Fetch origin** ثم إن ظهر **Pull origin** — اضغطه.  
  (هذا يمنع تضارب «أنا عندي كود والسيرفر عنده كود أحدث».)

### 3) بعد ما تعدّل ملفات في المشروع

1. في GitHub Desktop ستظهر الملفات المغيّرة على اليسار.
2. اختر الملفات المراد رفعها (أو كلها).
3. اكتب **Summary** للكومِت مثلاً: `fix: update checkout` أو `chore: docker setup`
4. **Commit to main**
5. **Push origin**

بعد الـ **Push**، إذا الموقع مربوط بـ **Cloudflare** من نفس الريبو، يبدأ **بناء تلقائي** (انتظر حتى يصبح Build ناجحًا).

## جزء ج — Supabase (قاعدة البيانات)

### 1) فتح المشروع

1. ادخل [supabase.com](https://supabase.com) → مشروعك (مثلاً اللي رابطه ينتهي بـ `.supabase.co`).

### 2) تشغيل الجداول (مرة واحدة)

1. من القائمة: **SQL** → **New query**
2. افتح على جهازك الملف (من مجلد `MUHRA JEWELRY`):

   `supabase\migrations\20250207120000_muhra_products_orders.sql`

3. انسخ **كل** محتوى الملف والصقه في محرر SQL في Supabase.
4. اضغط **Run**. يجب أن يظهر نجاح بدون خطأ.
5. من **Table Editor** تأكد أن الجداول **`products`** و **`orders`** ظهرت.

### 3) المفاتيح التي تحتاجها لاحقًا (من Settings → API)

- **Project URL** → يصير `NEXT_PUBLIC_SUPABASE_URL` على الاستضافة.
- **service_role** (Secret) → يصير `SUPABASE_SERVICE_ROLE_KEY` على الاستضافة فقط.  
  **لا ترسله لأحد ولا ترفعه للمعرض العام.**

## جزء د — Cloudflare (النشر والمتغيرات)

### 1) المشروع عندك اسمه غالبًا `muhra1`

- **Workers & Pages** → المشروع المربوط بـ GitHub `muhra1` → فرع **main**.

### 2) المتغيرات (Settings / Variables)

أضف (للبيئة **Production** على الأقل):

| الاسم | ملاحظة |
|--------|--------|
| `NEXT_PUBLIC_SUPABASE_URL` | نفس رابط المشروع من Supabase |
| `SUPABASE_SERVICE_ROLE_KEY` | **Secret** — من API كـ service_role |
| `STAFF_COOKIE_SECRET` | **Secret** — أي نص عشوائي طويل (16 حرف فأكثر) |
| `STAFF_USERNAME` | مثلاً `staff` — يجب أن يطابق ما يستخدمه `/api/staff/session` |
| `STAFF_PASSWORD` | نفس كلمة المرور التي تدخل بها في `/staff/login` إذا كنت تستخدم القيم الافتراضية |

**مهم:** بعد حفظ المتغيرات، نفّذ **إعادة نشر** (Redeploy) إذا المنصة لا تعيد البناء تلقائيًا.

### 3) الدومين `muhrajewelry.com`

- من نفس مشروع Cloudflare: **Custom domains** وأضف الدومين واتبع التعليمات.
- DNS إما عند Cloudflare (Nameservers) أو سجلات عند الجهار (GoDaddy) كما يقترح المعالج.

## جزء هـ — التحقق أن «كلشي شغال» قبل التسليم

افتح من **الهاتف أو متصفح خاص** (لتتفادى كاش قديم):

1. `https://muhrajewelry.com/api/catalog/products`  
   - متوقع: `{"products":[ ... ]}`  
   - إذا: `{"error":"backend_not_configured"}` → ناقص متغيرات على Cloudflare أو النشر قديم.

2. الصفحة الرئيسية + **المنتجات**.

3. **/staff/login** → دخول → من تبويب **Products** أضف منتجًا → رجع للموقع العام وتأكد أنه يظهر بعد تحديث الصفحة.

4. **طلب تجريبي** (COD) حتى صفحة النجاح.
